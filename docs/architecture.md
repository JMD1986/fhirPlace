# System Architecture

## Overview

fhirPlace is a two-tier web application that lets clinicians and hospital IT teams search, browse, and analyse synthetic FHIR R4 patient records. It ships with a bundled dataset generated by [Synthea](https://github.com/synthetichealth/synthea); in production it can also authenticate against any SMART on FHIR–compliant EHR.

### Development layout

| Process | Default port | Started by |
|---------|-------------|-----------|
| Vite dev server (React SPA) | 5173 | `npm run dev` |
| Express API server | 5001 | `npm run server` |
| Both together | — | `npm run dev:all` |

### Production layout (Docker)

```
Internet ──► Reverse proxy / CDN (443)
                 ├── /          ──► Static SPA (dist/)
                 └── /api/ /fhir/ ──► API container (5001)
```

The Dockerfile uses a two-stage build: only `server.js`, `node_modules` (production deps), and `public/synthea/` are included in the final image.

## SMART on FHIR authentication flow

fhirPlace supports two launch modes as specified by the [SMART App Launch 2.0](https://hl7.org/fhir/smart-app-launch/) specification.

### Standalone launch (default in development)

```
User clicks "Sign in"
  │
  ├─► LaunchPage reads VITE_SMART_ISS (or user-supplied server URL)
  │
  ├─► FHIR.oauth2.authorize() redirects browser to EHR authorization endpoint
  │     • sends client_id, scope, redirect_uri, PKCE code_challenge
  │
  ├─► User authenticates with EHR
  │
  ├─► EHR redirects to /callback?code=…&state=…
  │
  ├─► CallbackPage calls FHIR.oauth2.ready()
  │     • exchanges authorization code for tokens
  │     • populates AuthContext with AppUser + fhirclient Client
  │
  └─► App displays patient/provider views
```

### EHR-embedded launch

```
EHR opens fhirPlace with ?iss=<server>&launch=<token> in the URL
  │
  └─► Same PKCE flow, but launch token is forwarded to the token endpoint
      so the EHR can pre-populate patient/encounter context
```

## Data flow — patient search

```
PatientSearch component
  │  name typed
  ▼
patientApi.search(URLSearchParams)
  │  GET /fhir/Patient?name=&_count=&_offset=
  ▼
server.js – search against patientListCache (in-memory Map)
  │  filter by family/given/full name (case-insensitive)
  ▼
Returns FHIR Bundle (searchset) – bundled list of Patient resources
  ▼
SearchResults renders patient cards
```

## Security controls

| Control | Mechanism |
|---------|-----------|
| HTTP security headers | `helmet` (CSP, HSTS, X-Frame-Options, Referrer-Policy) applied in both `server.js` and Vite dev server config |
| CORS | Explicit allowlist (`http://localhost:5173`, `http://localhost:3000`); update for production origin |
| Authentication | SMART on FHIR (OAuth 2.0 + PKCE) via `fhirclient` library |
| Non-root container | Docker `USER node` directive |
| Dependency scanning | `npm audit --audit-level=high` runs in CI on every PR |
| Secret management | All secrets in `.env` (git-ignored); never baked into the image |
