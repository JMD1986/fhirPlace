# Troubleshooting FAQ

Answers to the questions most commonly asked by IT staff evaluating or deploying fhirPlace.

---

## Installation & startup

### The app opens but shows "Cache not ready" on all requests

The Express server indexes FHIR JSON files on startup. If `public/synthea/fhir/` is empty or missing, the cache never populates.

**Fix:**

```sh
npm run synthea:setup
npm run synthea:run -- -p 50
npm run synthea:manifest
```

Then restart the server (`npm run server`). Watch the startup log for the line:
`🏥 FHIR Patient Server running on http://localhost:5001`

---

### `npm run dev:all` starts but the browser shows a blank page or network errors

The React app on port 5173 can't reach the API on port 5001.

**Checks:**

1. Is the server running? Visit `http://localhost:5001/api/health` directly in your browser.
2. Is something else using port 5001? `netstat -an | findstr 5001` (Windows) or `lsof -i :5001` (Mac/Linux).
3. Is there a firewall rule blocking localhost traffic? Temporarily disable and retry.
4. Check the browser's Network tab for CORS errors — the origin of the dev server (`http://localhost:5173`) must be in the `corsOptions` allowlist inside `server.js`.

---

### `npm install` fails with ERESOLVE / peer dependency errors

Make sure you are using Node.js 20+:

```sh
node -v   # should be v20.x.x or higher
```

If you have multiple Node versions, use [nvm](https://github.com/nvm-sh/nvm) or [nvm-windows](https://github.com/coreybutler/nvm-windows) to switch:

```sh
nvm install 20
nvm use 20
npm install
```

---

### Synthea setup fails: "Java not found" or "java: command not found"

Synthea requires Java 11 or newer (full JDK, not just the JRE).

**Fix:** Download from [Adoptium](https://adoptium.net/) and ensure `java` is on your `PATH`:

```sh
java -version   # should print version 11 or higher
```

## Authentication

### Clicking "Sign in" redirects to the auth server but then returns a `redirect_uri_mismatch` error

The redirect URI sent in the OAuth request does not match what is registered in the SMART sandbox.

**Fix:** Ensure `VITE_SMART_REDIRECT_URI` (or the dynamically derived `window.location.origin + "/callback"`) exactly matches the registered URI, including scheme and port.

- Development: `http://localhost:5173/callback`
- Production: `https://fhirplace.example.com/callback`

---

### After the auth redirect the app shows "Ready error" or stays on the callback page

This usually means the authorization code exchange failed (expired code, state mismatch, or PKCE failure).

**Steps:**

1. Clear browser storage: open DevTools → Application → Local Storage → clear all entries for `localhost`.
2. Retry the login flow.
3. If it still fails, check the browser console for the full error from `fhirclient`.

---

### The app works in development but shows auth errors after deploying to production

Common causes:

- `VITE_SMART_ISS` points to the dev sandbox (`r4.smarthealthit.org`) instead of the production EHR.
- `VITE_SMART_CLIENT_ID` has not been registered with the production EHR — contact your EHR vendor.
- `VITE_API_BASE` is `http://` instead of `https://` — the app blocks startup and logs a security warning.
- The redirect URI registered with the EHR does not match the production domain.

## FHIR data

### Patient search returns no results

1. Confirm the server is fully loaded: `GET /api/patients-count` should return a non-zero count.
2. Check the spelling — the search is a **substring** match against the full name field. Try a shorter search term.
3. If using a remote FHIR server (`VITE_SMART_ISS`), the server may require a minimum of 3 characters before returning results — this is a server-side restriction.

---

### Encounters or clinical data are missing for a patient

The dataset was generated by Synthea. Some patients have few or no encounters (e.g. newborns with only well-child visits). This is not a bug.

If you need richer data, regenerate with more patients:

```sh
npm run synthea:run -- -p 500 -o public/synthea/fhir
npm run synthea:manifest
```

---

### The browser shows CORS errors when calling `/fhir/*` endpoints

The production server's `corsOptions.origin` array must include your deployed SPA origin. Edit `server.js`:

```js
const corsOptions = {
  origin: ["https://fhirplace.example.com"],
  credentials: true,
};
```

Rebuild and redeploy the API container.

---

### NPI Registry (provider search) always returns 0 results

The NPPES proxy (`GET /api/nppes`) calls `npiregistry.cms.hhs.gov`. If the API server cannot reach that domain (e.g. egress firewall blocks it), it returns a 502 error.

**Fix:** Add `npiregistry.cms.hhs.gov` port 443 to your outbound firewall allowlist. See [Network Requirements](./network-requirements.md).

## Docker / production

### The container starts but returns 503 on all `/fhir/*` routes

The container's `public/synthea/fhir/` directory is empty.

**Fix:** The Dockerfile copies `public/synthea` at build time (`COPY public/synthea ./public/synthea`). Make sure patient JSON files are present before running `docker build`.

---

### The container exits immediately with `Error: Cannot find module`

The two-stage Docker build copies only `node_modules` from the `deps` stage. If you modified `package.json` without running `npm ci` first, the modules may be out of sync.

**Fix:**

```sh
docker build --no-cache -t fhirplace-api .
```

---

### Port 5001 is already in use on the host

Remap the container port:

```sh
docker run -p 5002:5001 fhirplace-api
```

Then update your reverse proxy to forward to port 5002.

## CI / GitHub Actions

### The `audit dependencies` step fails

`npm audit` found a **high or critical** vulnerability in a dependency.

1. Run `npm audit` locally and read the advisory.
2. Update the affected package: `npm update <package>` or bump the version in `package.json`.
3. If no fix is available yet, consider adding an `npm audit --ignore-vulnerability <id>` exception with a documented comment and a remediation deadline.

---

### The Docker build step in CI fails

Check that `public/synthea/fhir/` contains at least one `.json` file before the `docker build` command runs. The CI workflow does not generate Synthea data — pre-generated files must be committed (or fetched from a release artifact) before the image is built.
